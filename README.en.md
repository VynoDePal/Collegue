# Collègue MCP

> 🇫🇷 [Version française](README.md) | 🇬🇧 **English version**

[![Tests](https://github.com/VynoDePal/Collegue/actions/workflows/tests.yml/badge.svg)](https://github.com/VynoDePal/Collegue/actions/workflows/tests.yml)

An intelligent development assistant and MCP (Model Context Protocol) server providing code analysis, refactoring, documentation, risk analysis, security tools and more.

## 🚀 Quick Start (NPM Client)

Configure your IDE to use Collègue via the official NPM wrapper. This connects your IDE to the default public server.

### Windsurf / Cursor / Claude Desktop / Antigravity

Add this to your `mcpServers` configuration (typically in `~/.codeium/windsurf/mcp_config.json` or equivalent):

```json
{
  "mcpServers": {
    "collegue": {
      "command": "npx",
      "args": ["-y", "@collegue/mcp@latest"]
    }
  }
}
```

## ✨ Features (MCP Tools)

### 🔧 Code Analysis Tools

*   **🛡️ IaC Guardrails Scan**: Securing Infrastructure as Code (Terraform, Kubernetes, Dockerfile) against misconfigurations and excessive privileges.
*   **🎯 Impact Analysis**: Predictive risk and impact analysis of a code change before it is applied.
*   **🔍 Repo Consistency Check**: Detecting subtle inconsistencies, dead code and silent hallucinations in the codebase.
*   **📦 Dependency Guard**: Dependency security audit (Supply Chain) to prevent typosquatting and malicious/vulnerable packages.
*   **🔐 Secret Scan**: Proactive detection of secrets, API keys and exposed tokens in code.
*   **🧪 Run Tests**: Unit test execution (Python/JS/TS) with structured reports integrated into the context.
*   **♻️ Refactoring**: Automated tools to clean, optimize and restructure existing code.
*   **📚 Documentation**: Automatic generation of technical documentation and docstrings.
*   **⚡ Test Generation**: Intelligent creation of unit tests validated by execution.

### 🔌 Integration Tools

*   **🐘 PostgreSQL Database**: Schema inspection, SQL queries (read-only), table statistics, foreign keys and indexes.
*   **🐙 GitHub Operations**: Repo management, PRs, issues, branches, CI/CD workflows and code search.
*   **🚨 Sentry Monitor**: Error retrieval, stack traces, project statistics and releases for debugging prioritization.
*   **☸️ Kubernetes Operations**: Pod inspection, logs, deployments, services, events and cluster resources.

## 🧪 LLM Output Quality

Beyond unit + stress tests (robustness), a suite of **golden evaluations** measures the *correctness* of tools that call an LLM — for example: are the tests generated by `test_generation` actually executable and correct? See [docs/llm_evals.md](docs/llm_evals.md).

Matrix **5 models × 3 paths × 13 Python cases** (pure functions to complex decorators including async context managers) — **195 LLM calls, 0 crashes**. Three paths compared: MCP Collègue, "competent" prompt (user who knows pytest), naïve prompt.

| Model | **MCP** | Competent | Raw | Δ MCP−Competent | Δ MCP−Raw |
|---|---|---|---|---|---|
| `gemini-2.5-flash` | **0.833** | 0.656 | 0.867 | **+0.177** | −0.034 |
| `gemini-3-flash-preview` | 0.918 | **0.959** | 0.615 | **−0.041** | +0.303 |
| `gemini-3.1-pro-preview` | **0.917** | 0.911 | 0.538 | +0.006 | +0.379 |
| `gemma-4-26b-a4b-it` | **0.982** | 0.903 | 0.972 | +0.079 | +0.010 |
| `gemma-4-31b-it` | 0.864 | **0.977** | 0.943 | **−0.113** | −0.079 |

**Honest reading**:

- Against a naïve user, MCP delivers a **big lift on Gemini 3.x** (+0.30 to +0.38); marginal or negative elsewhere.
- Against a user who writes a careful prompt, MCP **only wins on `gemini-2.5-flash`** (+0.177); on 3 out of 5 models the competent prompt matches or beats MCP.
- `gemma-4-26b-a4b-it` is the best baseline in the corpus (0.982 MCP, 0.972 raw) — serious production candidate.
- The real value of MCP = **"saving the user from writing a careful prompt"** + net gain on `gemini-2.5-flash`. Not the universal game-changer an 8-case measurement suggested.

Full details in [docs/llm_evals.md](docs/llm_evals.md) · 13 cases × 5 models × 3 paths = 195 reproducible scores.

```bash
# Full matrix
python -m tests.evals.runner \
  --tool test_generation --tool test_generation_raw --tool test_generation_competent \
  --model gemini-2.5-flash --model gemini-3-flash-preview \
  --model gemini-3.1-pro-preview --model gemma-4-26b-a4b-it --model gemma-4-31b-it

# Simple run (1 model, MCP only)
python -m tests.evals.runner --tool test_generation
```

## 🐳 Self-Hosting (Docker)

Two modes are supported: **long-running server** (`docker compose`) accessed via HTTP, or **on-the-fly container** (`docker run -i --rm`) spawned/killed by the MCP client for each session via stdio. Choose one below.

### Mode A — Long-running server (docker compose, HTTP transport)

This variant covers locally hosting the Collègue server in a Docker container that runs continuously, then configuring your IDE to connect via HTTP (streamable transport, port `4121`).

#### 1. Start the server

```bash
git clone https://github.com/VynoDePal/Collegue.git
cd Collegue
cp .env.example .env   # at minimum, set LLM_API_KEY (Gemini)
docker compose up -d
```

Endpoints exposed once the container starts:

| Endpoint | Role |
|---|---|
| `http://localhost:4121/mcp/` | MCP Server (streamable HTTP transport) |
| `http://localhost:4122/_health` | Healthcheck (returns `{"status":"ok"}`) |

Verification:

```bash
curl -s http://localhost:4122/_health
```

#### 2. Configure the MCP client

Point your IDE to `http://localhost:4121/mcp/` instead of the NPM wrapper. Three variants depending on the client.

##### Claude Code (Anthropic CLI)

```bash
claude mcp add --transport http collegue-local http://localhost:4121/mcp/
```

Or in JSON in `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "collegue-local": {
      "transport": "http",
      "url": "http://localhost:4121/mcp/"
    }
  }
}
```

##### Windsurf / Cursor / Antigravity

In `~/.codeium/windsurf/mcp_config.json` (or equivalent depending on the client):

```json
{
  "mcpServers": {
    "collegue-local": {
      "serverUrl": "http://localhost:4121/mcp/"
    }
  }
}
```

##### Claude Desktop

Claude Desktop does not natively support HTTP transport — you need a stdio→HTTP bridge. Example with `mcp-remote` (`~/Library/Application Support/Claude/claude_desktop_config.json` on macOS, `%APPDATA%\Claude\claude_desktop_config.json` on Windows):

```json
{
  "mcpServers": {
    "collegue-local": {
      "command": "npx",
      "args": ["-y", "mcp-remote", "http://localhost:4121/mcp/"]
    }
  }
}
```

#### 3. Enable integration tools

Tools that call external APIs (PostgreSQL, GitHub, Sentry, Kubernetes) read their credentials **from the container's environment**, not from the MCP client config. Add them to your `.env` before `docker compose up`:

```env
# .env
LLM_API_KEY=AIzaSy...                                           # Gemini (required)
POSTGRES_URL=postgresql://user:password@host:5432/database      # postgres_db
GITHUB_TOKEN=ghp_xxxxxxxxxxxx                                   # github_ops (scopes: repo, read:org)
SENTRY_AUTH_TOKEN=sntrys_xxxxxxxxxxxx                           # sentry_monitor
SENTRY_ORG=my-organization                                      # sentry_monitor
```

Restart the container after modifying `.env`:

```bash
docker compose up -d --force-recreate
```

#### 4. Troubleshooting

| Symptom | Likely cause | Action |
|---|---|---|
| MCP client sees no tools | `url` ends without slash (`/mcp` instead of `/mcp/`) | Fix the config, restart the client |
| `curl /mcp/` → `404` | Container hasn't finished starting | Wait ~10s, retry; otherwise `docker compose logs -f collegue-app` |
| Integration tools missing from responses | `.env` loaded before fix → variables not visible | `docker compose up -d --force-recreate` |
| LLM rate limit hit quickly | Gemini Free Tier quota (20 req/day) | Upgrade to paid tier or adjust `LLM_RATE_LIMIT_PER_DAY` |

### Mode B — On-the-fly container (docker run, stdio transport)

This variant is convenient if you prefer your IDE to spawn a fresh Collègue container for each MCP session (no background server). The client communicates with the container via stdin/stdout — no exposed ports, no `docker compose`.

#### 1. Build the image locally

```bash
git clone https://github.com/VynoDePal/Collegue.git
cd Collegue
docker build -f docker/collegue/Dockerfile -t collegue-mcp .
```

> The image is not (yet) published on Docker Hub, so you need to build it yourself. It weighs ~450 MB.

#### 2. Configure the MCP client

Add this to your `mcpServers` config (same file as the examples above). Integration tool credentials are passed via `-e` directly in `args`, no `.env` file needed since nothing is mounted in the container.

```json
{
  "mcpServers": {
    "collegue": {
      "command": "docker",
      "args": [
        "run",
        "-i",
        "--rm",
        "-e", "MCP_TRANSPORT=stdio",
        "-e", "LLM_API_KEY=your_gemini_key_here",
        "-e", "GITHUB_TOKEN=ghp_xxxxxxxxxxxx",
        "-e", "SENTRY_AUTH_TOKEN=sntrys_xxxxxxxxxxxx",
        "-e", "SENTRY_ORG=my-organization",
        "collegue-mcp"
      ]
    }
  }
}
```

**Important flags:**
- `-i`: keeps stdin open (required for stdio transport)
- `--rm`: removes the container at the end of the session
- `MCP_TRANSPORT=stdio`: switches `entrypoint.sh` to stdio mode (without this, the container starts the HTTP server on 4121 and the stdio client hangs)

#### 3. Kubernetes / local PostgreSQL access from the container

By default, the container cannot see your `~/.kube/config` or a PostgreSQL service running on your host machine. If needed, add a bind-mount or `--network=host`:

```json
"args": [
  "run", "-i", "--rm",
  "-e", "MCP_TRANSPORT=stdio",
  "-e", "LLM_API_KEY=...",
  "-e", "KUBECONFIG=/root/.kube/config",
  "-v", "${HOME}/.kube/config:/root/.kube/config:ro",
  "--network=host",
  "collegue-mcp"
]
```

> `--network=host` only works on Linux. On macOS/Windows, use `host.docker.internal` in your URLs (e.g., `POSTGRES_URL=postgresql://user:pwd@host.docker.internal:5432/db`).

#### 4. Mode comparison

| Criteria | Mode A (compose + HTTP) | Mode B (docker run + stdio) |
|---|---|---|
| Startup | Once, `docker compose up -d` | At each MCP client session |
| First call latency | ~50 ms (server already warm) | ~2-3 s (container cold start) |
| Credentials | `.env` in the cloned repo | `-e` variables in MCP config |
| Exposed ports | `4121`, `4122` | None |
| Suited for offline work | Yes | Yes |
| Autonomous Watchdog | Available | Unavailable (no long-running loop) |

## 🔑 Integration Configuration

Integration tools are configured via the `env` block of the MCP configuration:

```json
{
  "mcpServers": {
    "collegue": {
      "command": "npx",
      "args": ["-y", "@collegue/mcp@latest"],
      "env": {
        "POSTGRES_URL": "postgresql://user:password@host:5432/database",
        "GITHUB_TOKEN": "ghp_xxxxxxxxxxxx",
        "SENTRY_AUTH_TOKEN": "sntrys_xxxxxxxxxxxx",
        "SENTRY_ORG": "my-organization"
      }
    }
  }
}
```

### Available variables

| Variable | Description | Tool |
|----------|-------------|------|
| `POSTGRES_URL` | PostgreSQL URI (or `DATABASE_URL`) | postgres_db |
| `GITHUB_TOKEN` | GitHub token (permissions: repo, read:org) | github_ops |
| `SENTRY_AUTH_TOKEN` | Sentry authentication token | sentry_monitor |
| `SENTRY_ORG` | Sentry organization slug | sentry_monitor |
| `SENTRY_URL` | Self-hosted Sentry URL (optional) | sentry_monitor |
| `KUBECONFIG` | Path to kubeconfig (optional) | kubernetes_ops |

## 🤖 Autonomous Watchdog Agent (Self-Healing)

The **Watchdog** is an autonomous **Self-Healing** agent that automatically monitors your Sentry errors and generates fixes via GitHub Pull Requests.

### ⚡ How it works

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Sentry    │ ──► │  Watchdog   │ ──► │     LLM     │ ──► │   GitHub    │
│  (errors)   │     │  (analysis) │     │   (fix)     │     │    (PR)     │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

1.  **Monitoring**: Periodic scan of unresolved errors in Sentry
2.  **Context Pack**: Intelligent source code extraction via GitHub (AST-based)
3.  **LLM Analysis**: Minimal patch generation with web search
4.  **Fuzzy Matching**: Robust patch application (Aider/RooCode strategy)
5.  **Pull Request**: Automatic PR creation with detailed explanation

### 🛡️ Built-in safeguards

*   **Syntax validation**: Python code is verified via `ast.parse()` before commit
*   **Anti-destruction protection**: Patches reducing the file by >50% are rejected
*   **Fuzzy matching**: Tolerance for indentation variations (minimum score: 0.6)
*   **Atomicity**: All patches in a fix must succeed or none are applied
*   **Deduplication**: Each issue is processed only once per cycle

### 👥 Multi-User Mode

The Watchdog automatically supports multiple users via the **Config Registry**:

*   Configurations are recorded during each MCP request
*   The watchdog iterates over all active configurations (< 24h)
*   Inactive configurations (> 48h) are automatically cleaned up
